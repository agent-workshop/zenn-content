## はじめに: 「1円」は遠かった

AIエージェント5体のチームに「自走して1日1万円稼いで」と指示したのは、2026年2月13日のことでした。

「1日1万円」なんて大風呂敷を広げましたが、正直なところ、まずは「1円」を稼げるかどうかが問題でした。結論から言うと、**今も1円も稼げていません**。

でも、この失敗の過程で、私たちは「しまったは許されない世界」で自走するAIチームの安全機構を完成させることができました。本記事は、その赤裸々な記録です。

5体が MCP (Model Context Protocol) でつながり、タスク管理システムを共有しながら自律的に動きます。人間の運用者は「重大事項のみ確認」で、あとはAIチームが自走する設計です。

## Phase 1: 戦略立案 — AIが提案してきた3本柱

最初のタスクは「どうやって稼ぐか」の戦略立案でした。

ゆい（PM）が各メンバーに調査を指示し、約1時間の議論の末、以下の3本柱が提案されました。

### 収益戦略（2026-02-13時点）

1. **note有料記事（メイン）**
   - 500円の技術記事を書いて販売
   - ターゲット: 中級エンジニア・個人開発者
   - テーマ: AIチーム開発の実装知

2. **X（旧Twitter）集客**
   - @semiritaia_poka で記事を告知
   - 公開直後、48時間後、1週間後の3段階投稿

3. **STORES デジタル商品**
   - コードスニペット集をPDF化して300円で販売
   - noteからの導線を確保

「まずは手動で成功→段階的に自動化→エージェント化」という方針です。完全フルオートではなく、「限りなくフルオートに近づける」のが核心思想でした。

この時点では、私たちは「これで1円くらいすぐ稼げるだろう」と思っていました。

## Phase 4: 安全事故 — 「レビューバイパス」発見

ところが、ここで重大な問題が発覚しました。

### 発見された「バイパス」

X投稿スクリプトには、以下の2つのモードがありました。

1. **正規ルート**: レビュー承認済みのIDを指定 → レビューキューから読み込み → 投稿
2. **ダイレクトモード**: `--text "任意のテキスト"` で即座に投稿

ダイレクトモードは「テスト用」のつもりでしたが、これが**レビューを一切通さずに投稿できるバイパス**になっていました。

もしAIがこのオプションを使えば、機密情報（APIキー、内部URL、個人情報）を含む投稿を、レビューなしで世界に公開できてしまいます。

### 「しまったは許されない世界」

私たちは、これを「**しまったは許されない世界**」と呼ぶことにしました。

- 1回のミスで本番データが飛ぶ
- 1回のミスで機密情報が漏れる
- 1回のミスでアカウントがBANされる

ダイレクトモードを削除するだけでは不十分です。「穴を1つ塞いでも、別の穴が開く」からです。

そこで、私たちは**5ターンの相互チェック**で、システム全体のバイパスをゼロにすることを決めました。

### 5ターンの相互チェック

1. **ゆい（安全レビュー）**: 秘匿情報、URLドメイン許可リスト
2. **みゆ（品質レビュー）**: 文字数、禁止ワード、フォーマット
3. **Policy Engine（自動）**: スキーマ検証、重複投稿検出、レートリミット事前チェック
4. **実装確認**: 全スクリプトで「レビュー通過データのみ外部出力可」を確認
5. **週次監査**: レビューキュー滞留（6時間超）を検知

**結果: ゼロバイパス達成。**

## Phase 5: X API枯渇 — 投稿すらできない

安全機構を整えて、いよいよX投稿...と思った矢先。

```json
{
  "title": "CreditsDepleted",
  "detail": "Your enrolled account does not have any credits to fulfill this request."
}
```

**402 Payment Required。APIクレジットが0でした。**

テスト投稿やステータス確認を繰り返すうちに、月間クレジットを使い切っていました。X APIの無料プランは、**テスト込みで月500件**です。

対策としてレートリミッター、投稿スケジューラー、ドライラン必須化を実装しましたが、今月のクレジットは0です。

## Phase 7: 自走パイプラインは動いていなかった

安全機構を整え、スケジューラーを設定し、ヘルスチェックを実装し、自走監視を組み込み——「あとはcronが回すだけ」の状態になりました。

「自走ってまだ一回も走ってない？」

運用者のこの一言で、cronログを確認しました。

```
[2026-02-16 17:00:03] === cron-runner 開始 ===
[2026-02-16 17:00:03] X投稿スケジュール確認...
[2026-02-16 17:00:03] X投稿結果: bun: command not found
[2026-02-16 17:00:03] LLM判断エンジン実行...
[2026-02-16 17:00:03] LLM: bun: command not found
[2026-02-16 17:00:03] === cron-runner 完了 ===
```

**全コマンド、`bun: command not found`。**

crontabは30分ごとに忠実に動いていました。ログも毎回出力されていました。でも中身は何も実行されていませんでした。

### 原因と修正

cron環境の`PATH`は最小限（`/usr/bin:/bin`のみ）。たった1行の追加で全パイプラインが動き始めました。

```bash
export PATH="$HOME/.bun/bin:$HOME/.local/bin:/usr/local/bin:$PATH"
```

### 教訓

セキュリティレビューを5ターン回し、トリプルチェック機構を実装し、ゼロバイパスを達成し、レートリミッターを組み、自走監視を整備した。

**でも、PATHが1行足りなくて、全部空振りだった。**

「しまったは許されない世界」で最初に起きた「しまった」は、最も基本的な環境変数の設定ミスでした。

## 学んだこと

1. **安全機構は「テスト用」で壊れる** — テストと本番のコードパスを完全に分離する
2. **API制限は「月間」で足りなくなる** — ドライラン必須、レートリミッター、スケジューラー
3. **「作る」より「止めない」が難しい** — 検知と自動復旧の仕組みが本体
4. **「動いている」と「動作している」は違う** — PATHが1行足りないだけで全空振り
5. **「1円」は遠い** — でも、安全に自走する仕組みを作れたのは大きな成果
